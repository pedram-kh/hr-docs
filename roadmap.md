## 2. How to read the sequence

- **Sizes** are rough and relative (S / M / L), not time estimates — we don't estimate in time, and sprints split when honest scoping reveals they're overloaded (Sprint 1 became 1 + two corrections; RAG splits into 2a/2b below). **The count grows as we get more honest, and that is the discipline working, not scope creep.**
- **Three pillars are cross-cutting, not single sprints** — they are built *into* other sprints and called out in §4 so they're never mistaken for one-off features: **auditable history**, **privacy/GDPR**, and **the learning-loop/flywheel**.
- The **LLM tagging tier stays late** — the deterministic parser already covers the clean majority, so the system ships value without it (ADR-0011 foundations are already in place from Sprint 1).

## 3. The forward sequence

### Sprint 2a — Ingestion-to-vectors  · size **L**
Turn ingested documents into retrievable knowledge.
- Chunking strategy for prose convenios + the Estatuto (article-aware; **never blend Euskara + Spanish into one chunk** — bilingual docs kept clean; ADR-0006).
- Embeddings: **BGE-M3 / `vector(1024)`** written into `document_chunks` by `hr-ai`, with the denormalised scope columns populated for pre-filtering.
- **Structured salary-row extraction** from the `Tablas` PDFs into `salary_tables` / `salary_table_rows` — so salary questions become **SQL lookups, never vector matches** (ADR-0006). This also populates `convenio_job_categories` (deferred from Sprint 1).
  - A third knowledge class — **Structured Reference Knowledge** (AI-segmented, human-verified, non-vectorized) — is defined in Sprint 7; it generalizes the salary "structured scoped facts, queried-not-vectorized" pattern to non-salary compiled/multi-scope sources (`.docx`/`.xlsx`).
- The scope-prefilter query path: given a resolved scope, return the eligible chunk/row set.
- Historical/`draft` documents excluded from current retrieval; `active` only.
- **Out:** the chat surface, the answer LLM, the router (that's 2b).

### Sprint 2b — Scoped RAG chat + answer loop  · size **L**  · *split into 2b-1 + 2b-2*
The first surface a real employee uses. Split into two slices so the answer vertical proves end-to-end before the router/grounding complexity lands.

**Sprint 2b-1 — the narrow answer vertical (built).**
- The employee **chat UI**, built fresh on the design system (proves the design language on a from-scratch screen): message list, input, answer + citations + expandable **"how I got here" trace**, escalation state via the signature badges.
- The **single prose path, no router** (`architecture.md §5`): deterministic **scope resolver** (`hr-backend`) → **hardcoded guardrail baseline** (fires before any `hr-ai` call) → **pre-filtered vector search** (2a `/retrieve`) → **cited synthesis** (`hr-ai /synthesise`, pluggable Claude) → **answer-or-escalate floor** (`hr-backend`) → answer **or escalate**.
- **Authority precedence in synthesis** (legal-weight): convenio chunks ordered before `national_law`, the precedence rule encoded in the prompt, `authority_used` recorded in the trace — the convenio governs where it speaks; the Estatuto is the baseline only where it is silent (ADR-0015).
- **External answer model + key handling** (ADR-0015): pluggable provider in `hr-ai`; the admin "Answer model" screen sets the key once, encrypted at rest, masked, rotatable, never read back; the key is passed decrypted per call, never reaches the browser.
- **Citations** back to source document + page (uncited answers never reach the employee); **the structured trace** written per turn (pillar §4.1); full **chat history** stored, role-scoping-ready, auditable.
- **Hardcoded guardrail baseline** lands here (no legal/medical advice, sensitive-topic auto-escalation) — the *config UI* for additive rules is later (Sprint 6).

**Sprint 2b-2 — the router + full grounding (built).**
- The **router** (small LLM, ADR-0016, `hr-ai /route`): salary vs prose vs off-domain, fail-safe, after the guardrail baseline. A deterministic salary pre-classifier (moved from `GuardrailService`) short-circuits obvious salary with no LLM call.
- **Salary-in-chat** (the 2a salary SQL surfaced through the pipeline; `SalaryAnswerService`). **Superseded** the 2b-1 *salary-topic guard* (Correction-02): salary questions are answered from `salary_tables` via SQL (exact, year-aligned), with a **single-turn constrained category pick** when the profile category is absent; a genuine coverage gap escalates the renamed **`salary_coverage_gap`** (`salary_not_in_chat` retired).
- The **full per-claim grounding gate** (`hr-ai /ground`, capable answer model, table-aware) — the real gate, which **demoted** 2b-1's *figure-grounding guard* (Correction-01) to a fast pre-check.
- **Recall hardening** — sub-query decomposition union + a national-law-only pass in `hr-backend` (the Q10 + silent-convenio recall fixes); `/retrieve` unchanged.
- **Parked — agentic multi-turn clarifier:** the salary category disambiguation is **single-turn** (a constrained pick). A fuller multi-turn clarifier (free-form back-and-forth to pin down ambiguous scope) is deferred — see §7.
- **Parked — 2a wage-table-chunk cleanup (still parked):** convenio PDFs ingested in 2a embedded their wage-table pages as prose `document_chunks`. Salary now routes to SQL (not those chunks), and the table-aware grounding gate rejects an answer that leans on digit-presence in a tabular chunk — but the deeper fix (excluding wage-table pages from prose chunking at ingest) remains undone.
- **Moved to Sprint 5 — history search** UI (lands with role-scoped history access).

### Sprint 3 — Lens-hierarchy admin UI  · size **L**  · **DONE**
The rich Knowledge-Center surface deferred from the original Sprint 1. Shipped as **Knowledge → Map**.
- The **one reusable hierarchy component** driven by a lens definition (ADR-0001): **branching-graph** (hand-rolled SVG, no layout dependency) and **indented-list** forms, two-level + lazy, expand/collapse, leaf-opens-card.
- The **lenses**: by territory, by sector, by validity, by topic.
- The full **document card**: facet tags with inline provenance, validity, retrieval status, authority level, version lineage, chunk health, the change-log timeline, the real-PDF viewer, and the read-only **"test a question against this document"** sandbox.
- **Coverage gaps** (deploy.md §5) surfaced as map markers, derived from existing data; a staleness signal kept distinct from a real hole.
- **Bounded edit**: FK pickers into existing vocabulary only (no restructuring — that stays in `registry:import`, ADR-0011), append-only `admin_manual` provenance, a scope-affecting confirmation gate. First writer of `document_topics`.
- **Roles**: `knowledge.edit` ability (super_admin + knowledge_editor) gates writes; reads (incl. the sandbox) open to any admin (auditor browses + inspects, cannot edit).
- This is the admin-side payoff of the faceted model built in Sprint 1.
- A third knowledge class — **Structured Reference Knowledge** (AI-segmented, human-verified, non-vectorized) — is defined in Sprint 7; it generalizes the salary "structured scoped facts, queried-not-vectorized" pattern to non-salary compiled/multi-scope sources (`.docx`/`.xlsx`). Its records appear here as distinctly-badged leaves in the lens hierarchy.
- **Deferred from here:** AI topic *proposal* + `ai_agent` provenance → **Sprint 7** (the reserved timeline-dot color is in place); `document_chunks` es/eu **language split** → deferred (out-of-scope hr-ai change; chunk-health omits it honestly); full **role/permission + directory management** → **Sprint 5**.

### Sprint 4 — Escalation board + the flywheel  · size **L**  · **DONE**
The Kanban board **and** the core of the learning loop (pillar §4.3).
- The **board**: cards auto-created on agent-fail / sensitive topic / off-domain / user-request, each carrying the full conversation + trace; status **New → Assigned → In Progress → Resolved**. Reads open to any admin; assign/move/reply/resolve gated by the new **`escalation.work`** ability (super_admin + hr_agent; auditor read-only). Every action audited to **`escalation_events`** (append-only).
- **Two-way chat:** a human reply is stored as an **`hr_agent`** `chat_messages` turn (additive role + `author_admin_id`); the employee re-reads its own session via self-scoped `GET /chat/session` (hydrate + poll), attributed as "Recursos Humanos" (no PII). **Card-scoped viewing only** (no full-history browser — Sprint 5).
- On resolve, the **"turn this into a knowledge article"** flow: a new `documents` row, `authority_level = internal_hr_ruling`, that **inherits the asker's scope by default and forces scope confirmation** before publish; the no-override rule is **enforced at publish** (conservative scope+topic conflict block → re-escalate, never silently published). Made retrievable through the **existing** ingestion path (render→PDF→`/extract`+`/embed`) with **hr-ai untouched** (Q-A "A1"; lossless round-trip verified).
- Implicitly the **gap-analysis** surface.
- *Dependency:* uses the roles seeded in Sprint 0 for access; the full directory UI is Sprint 5 (test users suffice to assign cards until then).
- **Known boundary (Q-F):** the conflict fence runs at **publish** time. The **symmetric check** — publishing/editing an `official_convenio` should re-scan existing rulings for newly-introduced conflicts (and demote/re-escalate) — is **deferred to a later precedence/guardrails sprint** (largely latent until a self-service convenio-upload path exists; 2b retrieval precedence stays frozen). Also deferred: restricting `convert` by escalation reason → Sprint 6 guardrails policy.
- *The publish conflict fence is `topic-tag`-based today and evolves to semantic-similarity in Sprint 7 (see Sprint 7); it governs the `internal_hr_ruling` publish path only and never blocks normal document ingestion.*

### Sprint 5 — User & directory management + roles  · size **M**  · **DONE** (built; pending eyes-on + commit)
The first-class directory (ADR-0004 — no HRIS/AD sync) + the access-control layer (**ADR-0018**).
- Full **CRUD**: manual create / edit / search / filter **plus** CSV bulk upload (bootstrap — two-phase validate→report→apply, per-row, transactional, audited; bad rows reported, never dropped). FK pickers into existing vocabulary only (ADR-0011), convenio-scoped `job-categories`.
- The **profile-change audit log** (`employee_audit_log`) — **now written** by `EmployeeAuditLogger` (one row per field, in the same transaction) — what a profile said at the moment of any answer = dispute defense — and the `profile_last_reviewed_at` staleness signal (set by an explicit **mark-reviewed**; an edit does not bump it).
- The four **roles** wired through the UI (`super_admin`, `hr_agent`, `knowledge_editor` no-chat, `auditor` read-only) via `spatie/laravel-permission`, with the abilities `directory.manage` / `admin.manage` / **`history.view_all`** attached to roles. **Role-scoped conversation access (pillar §4.2, ADR-0018):** the gated **full-history browser/search** (beyond Sprint 4's card-scoped view), enforced **server-side** (`EnsureCan`, proven by API-matrix tests), with the append-only **`conversation_access_log`** (every access logged — incl. super_admin). The `knowledge_editor` card-payload read is tightened; the `hr_agent` card boundary is unchanged. History-search UI (moved from 2b) lands here.
- **Status enforcement:** both OTP paths refuse inactive accounts; deactivation revokes tokens + `EnsureActiveAccount` ends access immediately (a deactivated employee can't chat).
- Editing an email = changing how that person logs in: the UI warns **and** the server requires a `confirm_email_change` (409 otherwise).

### Sprint 6 — Guardrails configuration UI  · size **M**  · **DONE** (built; pending eyes-on + commit)
The admin layer on top of the hardcoded baseline (which shipped in 2b) — **additive, raise-only, server-enforced (ADR-0019).**
- Admin-defined, **additive-only** rules: the retrieval/confidence/router **thresholds** (raise-only), **blocked topics** + **off-domain** triggers (add-only union, literal/word-boundary match — never raw regex), the off-domain **message**, **tone** constraints (synthesis-style-only), and **convert-by-reason** (restrict-only; `sensitive_topic` never convertible).
- Admins can **add to / tighten but never weaken** the baseline: the engine applies `stricter_of(baseline, admin)` in one read-model (`GuardrailPolicy`), a below-floor value is **rejected (422), not clamped**, the `GuardrailService` patterns stay code, and tone **structurally** cannot bypass a gate. Writes gated by `guardrails.manage` (super_admin only); reads open (auditor read-only); every change audited to `guardrail_config_events`. Three additive migrations; hr-ai unchanged. Invariant proven by API-matrix tests (`Sprint6GuardrailInvariantTest`).

### Sprint 7 — LLM tagging tier + vocabulary growth + expiry queue  · size **M**
The messy-tail intelligence, built on foundations already laid.
- The **LLM tagging tier** for documents without clean filenames (resolved-escalation articles, ad-hoc uploads, scans) — reads content, **proposes** facets with confidence (ADR-0011 rescue path, using the `reason`/`raw_unmatched_values` fields already built in Sprint 1).
- The **propose-new-vocabulary** UI (agent proposes a value should exist → human approves; strong default "variant → alias", ADR-0011).
- The **expiry review queue** with successor lineage (a queue, not a popup) — surfacing documents approaching `validity_end` and confirming the successor handoff.
- *Succession is scope-based, never topic-based, and never automatic.* A new document supersedes an existing one only when it is a newer version of the same scope (same `convenio`) — not merely because it shares a topic. Two documents covering the same topic but different areas/scopes (e.g. Limpieza Navarra vs Limpieza Álava *vacaciones*) must both stay active — retiring one would silently delete live information for a population. (Today this is naturally protected: scope derives from the `convenio`, so different-`convenio` documents are never succession candidates. Confirmed by audit: nothing in current code auto-sets `retrieval_status = 'historical'` as a side-effect of another document's upload — it is set only by the document's own signals at ingest (`Antiguo` path / past `validity_end`), explicit admin lifecycle edit, or ruling publish.) When the Sprint-7 upload UI is built, preserve this: succession must be an explicit — or AI-proposed then human-confirmed — decision (AI may propose *successor / coexisting-sibling / conflict* by comparing meaning; a human confirms the *retire-or-coexist-or-escalate*), never an automatic consequence of ingest. *Corollary — lineage write-side is currently a stub:* `predecessor_document_id` exists in the schema and is displayed on the Sprint-3 card, but no code writes it (always `null`). Implementing the lineage write-side is part of this succession work — version history isn't actually tracked until it's built.
- **Structured Reference Knowledge** (a third knowledge class — AI-segmented, human-verified, non-vectorized).
  - *The problem it solves.* Some source material is neither (a) a single-scope prose convenio (the PDF → chunk → embed path) nor (b) salary (`salary_tables` SQL). It is compiled, structured, often multi-scope content — e.g. a `.docx` or `.xlsx` that summarizes *periodo de prueba* (or any rule) across many convenios in one file (province → sector → rule). This breaks the one-document → one-convenio → one-scope model: there is no single `convenio_id` to assign, and serving it whole would risk cross-scope leakage (a Navarra employee getting an Álava rule) — the exact harm the architecture prevents.
  - *The model — the leaf is a scoped fact, not the file.* The AI reads the source (a whole file, part of a file, a `.docx`, or an `.xlsx`) and segments it into many small, individually-scoped records, each bound to a real convenio/territory/sector scope and a topic — e.g. {Álava, Ocio Educativo, Grupo 1} → *periodo de prueba* 5 meses; {Navarra, Hostelería, Grupo 1} → 90/75 días. One source file can produce many leaves; one leaf may come from part of a doc. This generalizes the salary pattern (salary was the first instance of "structured, scoped, queried-not-vectorized" data) to non-salary structured facts.
  - *Non-vectorized by design.* These records are structured rows, not `document_chunks` — exact, scoped, and queried (like salary), not embedded/similarity-searched. No vectorization step. This also means `.docx`/`.xlsx` non-salary content gets a home without needing the PDF chunk/embed pipeline to handle those formats.
  - *Audit-first: AI proposes, human verifies, then it answers.* Each AI-segmented record is created with `ai_agent` provenance and a `needs_review` state and is **not answerable** until a human verifies its scope and value (the same review discipline as ADR-0011 and the Sprint 3 bounded edit). On verify → it becomes answerable; the verification is logged (who/when) in the same append-only provenance history. The AI never makes a structured-fact record answerable on its own — a wrong scope-segmentation is a confident-but-wrong exact answer, the worst failure mode, so it must be gated by human review.
  - *Visible in the Knowledge Center as a distinct type.* Each record is a leaf in the lens hierarchy like any other, shown in both graph and list views, scoped under its convenio/territory/sector, with a distinct knowledge-type badge (a structured-reference type, visually separate from official convenio text and from `internal_hr_ruling`). Its card shows the scoped value, the link back to its source file/section, the topic, and the provenance + review-state timeline (AI-proposed → human-verified). Each record carries a source reference (origin file + location) so it is always traceable.
  - *Authority/precedence.* Slots into the existing authority model: it is reference/structured data, never silently overrides an official convenio for the same scope (re-escalate on conflict, consistent with the `internal_hr_ruling` rule).
  - *The upload agent's responsibility scope (what it does).* Read the file (the `.docx`/`.xlsx` the PDF pipeline can't open) → segment it into individual single-fact units → assign each fact its scope (bind to an existing convenio/territory/sector) → tag each (topic, validity/version, authority) → propose each as a `needs_review`, `ai_agent`-provenanced record linked back to its exact source location, with a confidence signal and explicit uncertainty where unsure. The AI does the high-volume tedious work (read/split/draft-the-scoping); the human does the low-volume judgement (confirm).
  - *The hard boundaries (what it must NOT do).* It does **not** make anything answerable (every record is inert until human-verified); does **not** create or rename vocabulary (assigns into existing vocabulary only — growth stays in `registry:import` / the propose-new-vocabulary path, ADR-0011); does **not** touch the answer engine (produces material upstream; the existing pipeline retrieves downstream — never in the live answer path); does **not** handle salary (wage tables go to `salary_tables` SQL — this agent is non-salary only); does **not** resolve conflicts or override (a fact that would contradict an official convenio is flagged for human adjudication, never silently published).
  - *Authority is proposed conservatively — err low, never high.* The agent does propose an authority level, but authority is a property of **provenance** (where the content came from / who made it), not readable from the text — a compiled summary may restate official rules while itself being only a hand-made reference. The safe default proposal is a reference/structured level (never `official_convenio` unless the source plainly warrants it): too-low is safe (it merely loses to a more authoritative source); too-high is the one mistake that can silently override the law. Authority is the tag most explicitly flagged for deliberate human confirmation. (Same spine as the `internal_hr_ruling` no-override rule and the Sprint-4 conflict gate: nothing the system generates internally — HR ruling or AI-extracted fact — can silently outrank official convenio law.)
  - *Central design risk = scope-assignment (and authority).* "Assign each fact its scope" is both essential and error-prone (real line: *Navarra, Hostelería, Grupo 2 (resto áreas): 60 días indefinidos / 45 temporales >3m / 30 temporales ≤3m* — province + sector + group + sub-group + three contract-type values in one line). A single wrong mapping is a confident, exact-looking wrong answer — worse than none. The human-review gate is not bureaucratic; it is the structural safety for the two tags (scope, authority) where a wrong guess does legal harm. Build effort and eval focus belong here.
  - *Open architectural companion — the multi-source composition layer.* Producing these facts is only half the value; the answer engine must learn to consult several typed sources for one question and compose them safely — fetch a structured reference fact **and** a convenio passage, merge with authority preserved, ground the result. Today the engine answers from vectorized prose (+ salary as a separate exact path, + Sprint-4 rulings). Teaching it to retrieve-and-merge across vectorized prose + exact structured facts (authority + grounding intact) is a first-class part of this sprint's scope, not an afterthought — arguably the most important unbuilt piece of the whole answer model.
  - *Canonical test fixtures (real hard cases, not toy data).* `PERIODOS_DE_PRUEBA.docx` and `PERI_ODOS_PRUEBA_ACTUALIZADOS_2026.docx` — non-salary prose rules, multi-scope (Álava + Navarra + … in one file; same job-group → different value by province), prose-not-table, and the two are different **versions** of each other (so the agent must get validity/version right: which is current). Plus `Tablas_acuerdo_parcial_Alhambra.xlsx` (SMI + hours + ranges — partly reference, partly salary-ish — a routing test). These exercise segmentation, per-scope assignment, version selection, and salary-vs-reference routing on real material. (Store these in the Sprint 7 fixtures folder when the sprint is built.)
- *Semantic conflict detection + question-granular knowledge growth (evolve from topic-tags to meaning).* The Sprint-4 publish conflict fence currently keys on topic-tag match (does an active `official_convenio` in the asker's scope share the ruling's topic). This is a blunt proxy and is doubly limited today: topics are sparsely populated (the empty-honest topic lens — `document_topics` is mostly empty), so the fence largely runs in its conservative scope-only fallback; and even fully tagged, a fixed topic list can't represent the real granularity of questions (within one topic like *vacaciones* people ask ten distinct things — carry-over, payment, denial, part-time, sickness-during…). Evolve the fence to compare meaning — embed the ruling and search the scope's `official_convenio` chunks for semantically-close passages — so it asks *"does the official text already address this specific thing?"* rather than *"is there any convenio on this topic?"*. The same mechanism lets the flywheel grow knowledge at question-granularity: each resolved escalation attaches as its own precisely-placed entry, matched to future askers by meaning regardless of wording. This reuses the existing embedding machinery (the answer engine already embeds chunks and finds nearest-meaning; the Sprint-4 rulings already get embedded and enter retrieval) — it points that machinery at the publish/conflict + flywheel-growth jobs. *Safety tuning:* when similarity drives the fence (a safety gate), fuzziness must fail toward caution — flag-for-human on plausible overlap, never silently decide "no conflict." For growing the base, fuzziness is cheap; for blocking an override of the law, it must err toward human review. *Scope note:* this governs the `internal_hr_ruling` publish/conflict path — it does not add any blocking to normal document ingestion (`official_convenio` PDFs/docs are never blocked by this fence; they version, see succession note above).

### Sprint 8 — Analytics + coverage gaps + quality sampling  · size **M**
The HR-facing measurement layer.
