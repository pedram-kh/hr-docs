## Resolved open questions (apply exactly)

1. **Q1 — `directory.manage`: `super_admin` + `hr_agent`.** Agents do the day-to-day "employee transferred province" corrections (ADR-0004). Directory PII (name/scope) is a lower privacy tier than conversation content — acceptable. Reads of directory data gated to the same ability.
2. **Q2 — `admin.manage`: `super_admin` only.** Creating admins / granting `history.view_all` is the most privileged action; keep it at the top.
3. **Q3 — `history.view_all` is a spatie permission attached to roles** (not a role-name check) — composes with `EnsureCan`/`can()`, surfaces in `IdentityPresenter.abilities`, survives role edits.
4. **Q4 — CSV: validate → report → apply valid rows, each in its own transaction with its audit write; failures reported, never silently dropped.** (Not whole-file-atomic — one bad row must not kill a 1,500-row bootstrap.)
5. **Q5 — tighten the knowledge_editor card-conversation read: YES.** Gate the **conversation payload** of the Sprint-4 card detail behind `escalation.work OR history.view_all` (super_admin/hr_agent/auditor see it; knowledge_editor does not). The board **listing/meta** stays broadly readable; only the conversation **messages** are gated. **This must NOT loosen the `hr_agent` boundary** — `hr_agent` access stays keyed to `card.chat_session_id`, unchanged. Test **both** directions (knowledge_editor denied; hr_agent still works).
6. **Q6 — email change: add the server `confirm_email_change` 409 gate** (Sprint-3/4 confirm pattern) in addition to the UI warning. Email is the login key; the change must be deliberate, not click-through. Always audited regardless.
7. **Q7 — access-log granularity:** one `conversation_view` row per **opened** conversation; one lighter `history_search` row per search run. A search **listing** (snippets shown without opening) logs `history_search`, not per-employee `conversation_view` — **and the snippets must be short** (a brief match fragment, not enough to read the conversation's substance; if a snippet would expose substance, it needs per-employee logging). Keep snippets brief by design.
8. **Q8 — inactive-employee login: in scope, build it here.** Apply the same active-status refusal to the employee OTP path (a deactivated employee can't chat). It's the same one-line consistency fix as the admin-status enforcement — don't defer.
9. **Q9 — editing a profile does NOT bump `profile_last_reviewed_at`.** "Reviewed" is an explicit attestation via `mark-reviewed`, distinct from "edited", so the staleness signal stays honest.
10. **Q10 — history date filter: `last_activity_at` primary, `started_at` available.** Query-filterable at 1,500 employees; no denormalization.

## Non-negotiable (the legally load-bearing parts — build + prove)

- **ADR-0018 is required** (not optional). Write **ADR-0018 — Role-scoped conversation access (`history.view_all`) + conversation access logging**: the decision that (1) broad conversation access is a **deliberately-granted ability held by a designated few**, not automatic for any role; (2) **every conversation access is logged, including `super_admin`'s**; (3) **the server is the boundary** (the UI only hides). It sits on the identity/privacy axis beside ADR-0003/0004; Sprint 9 references it.
- **Server-side enforcement, proven by API tests (the acceptance proof — §6.2 step 9).** The sprint is **not done** until these pass (not "the UI looks right"): `hr_agent` → **403** on every `history.view_all` endpoint, **200** only on their own card's conversation; `knowledge_editor` → **403** on all conversation reads (incl. the now-tightened card payload); `auditor` + `super_admin` → **200** on history **and a `conversation_access_log` row written**; a **deactivated** admin → 401/403 (login refused + active-status gate + token revoked). Hitting a forbidden endpoint **directly** (not via the UI) must 403 — prove the server is the boundary.
- **Every History access writes `conversation_access_log` — including `super_admin`** (§5.2). No role is exempt from the log. This is the accountability safeguard that makes the History browser defensible.
- **Every directory change writes `employee_audit_log`** (one row per changed field), inside the same transaction as the employee write (the row and its audit never diverge); CSV-applied rows audited too.

- **Additive migrations only:** `conversation_access_log` (the one new table) + the `history.view_all` permission seed (idempotent data migration **and** `RoleSeeder` in lockstep, so prod `migrate` and fresh installs both get it). ## Hard constraints (carry)

- **Server-side enforcement, never UI-only.** Every history/conversation/directory/admin endpoint checks its ability on the server; hiding a nav item is not access control. The **`hr_agent` card-scoped boundary (Sprint 4) must not loosen** — only the knowledge_editor read is tightened (Q5).
- **`hr-backend` owns all writes + schema; additive migrations only.** Roles via `spatie/laravel-permission` (no hand-rolled tables). FK pickers into **existing** vocabulary only (no vocabulary creation — that's `registry:import`, ADR-0011).
- **No HRIS/AD sync** (ADR-0004), **no SSO** (ADR-0003 — OTP stays), **no answer-loop change** (history is read-only over existing data), **no GDPR encryption/retention** (Sprint 9 — but the access log + status enforcement start here).
- Reuse the **design system** + the existing admin shell + the `EnsureCan` pattern + `ConversationPresenter`/`TracePanel`/`CitationList`. No new visual primitives expected.
