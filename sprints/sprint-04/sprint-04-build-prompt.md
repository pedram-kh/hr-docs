## Resolved open questions (apply exactly)

- **Q-A — ruling → retrievable: A1 (render to a stored artifact; hr-ai untouched).** On publish, hr-backend renders the ruling body to a **generated PDF** in S3, sets `storage_path`, and runs the **existing** `/extract` + `chunks:embed --document={uuid}` path. Add `internal_hr_ruling` to `ChunksEmbed::IN_SCOPE_TYPES` (hr-backend code change, not a migration). **hr-ai is not modified.** **Required verification:** confirm the round-trip is **lossless** — the embedded chunk text matches the agent's typed `resolution_text` (no de-spacing/furniture mangling on the clean generated PDF). Report the before/after in `review.md`. **If (and only if) the clean PDF is mangled by the 2a heuristics, fall back to A2** (a tiny additive hr-ai inline-text branch that skips PDF preprocessing) and flag it — do not silently ship a mangled ruling.
- **Q-B — conflict granularity: conservative scope+topic block (safe direction).** The gate blocks on **presence** of an active `official_convenio` in the **same convenio scope sharing a topic** with the ruling — it does **not** attempt a semantic "do they actually disagree" comparison. Rationale: a safety fence must err toward blocking (a false block routes to a human — safe; a false "no conflict" is the exact harm). **Require the agent to assign a topic at publish** (reuse the Sprint-3 approved-topic picker) — this sharpens the gate and seeds the topic lens. **If no suitable topic exists**, allow publish but the gate **falls back to scope-only block** (any active official_convenio in the asker's convenio scope → block/route to human — over-blocks, the safe failure).
- **Q-C — author model: `role += 'hr_agent'` + nullable `author_admin_id` FK.** Add `hr_agent` to the `chat_messages.role` CHECK via the proven introspect-drop-readd idiom (the `…add_salary_coverage_gap…` pattern; reversible, idempotent) + a nullable `author_admin_id` FK → admins. An `hr_agent` message has **no trace, no citations** (a human turn, not a synthesized answer).
- **Q-D — employee sees the reply: session-load + polling (minimal).** Add the employee-auth `GET /chat/session` (self-scoped to the caller's own most-recent session; returns ordered messages incl. `hr_agent` ones, attributed as "Recursos Humanos" — **never** the admin's email/PII). `ChatScreen` hydrates on mount + polls (~20–30s while the tab is open / on focus). **No websockets, no notifications** this sprint.
- **Q-E — audit table: dedicated `escalation_events`** (append-only: `escalation_card_id`, `type` ∈ `status_change|assigned|replied|resolved|converted|publish_blocked`, `old_value`/`new_value`, `actor_id`, `note`, `created_at`). Do **not** overload `tag_events` (its `facet` is document-vocabulary).
- **Q-F — retrieval-precedence residual: accepted as a documented boundary** (do not fix now). The publish gate covers conflicts **at publish**; a convenio added/edited *later* that conflicts with an already-published ruling is **not** auto-demoted at retrieval (2b is frozen). Record this clearly in `architecture.md` as a known boundary, and add a **roadmap note**: the symmetric check — *publishing/editing an `official_convenio` should re-scan existing rulings for new conflicts* — belongs in a later precedence/guardrails sprint (and is largely latent until the self-service convenio-upload path exists). Do not touch 2b precedence.
- **Q-G — employee follow-up: normal loop, new card on re-escalation.** The employee keeps chatting in the same session (the card-scoped view shows the continued conversation since it's session-bound); a follow-up that independently escalates creates a **new** card. No auto-reopen/threading/notifications.
- **Q-H — auditor board access: read-only, yes.** `auditor` may browse the board + open cards + read the conversation/trace, but **cannot** assign/move/reply/resolve (gated by `escalation.work`, denied to auditor). Mirrors the Sprint-3 auditor posture.

## Two things to honour in the build (not new questions — guardrails on scope)

- **The guardrail baseline still fires first.** A published `internal_hr_ruling` on a sensitive topic cannot bypass the hardcoded sensitive-topic escalation — the 2b-1 guardrail fires **before** retrieval, so the ruling is simply unreachable for genuinely sensitive queries (correct). `convert` stays the agent's judgment (optional, not reason-gated this sprint); whether to **restrict** convert by `reason` is a Sprint-6 guardrails-policy matter — note it, don't build it.
- **The ruling is single-convenio scoped** (inherits the asker's convenio; territory/sector ride it). A national/multi-scope ruling is **out of scope** (that's Structured Reference Knowledge, Sprint 7). Erring narrow is safe.
