**The five plan-review catches — build these in (do not skip):**

9. **Page-furniture stripping is a never-blend correctness requirement, not just hygiene (catch 1).** Before column assignment, **classify each block by span**: a block wider than ~70% of page width is **full-width furniture** (masthead, resolution title, the per-page footer) → it is **not** assigned to a column. Strip repeating headers/footers (detect by repeated text and top/bottom y-bands across pages). This is mandatory because the real Gipuzkoa footer is itself bilingual (`…Gipuzkoako Aldizkari Ofiziala / Boletín Oficial de Gipuzkoa…`) — leaving it in would create a **blended-language chunk**, which ADR-0013 forbids. Cluster `x_mid` only over the remaining **column-width** blocks for the two-column eu/es split. Report, per document, how many blocks were stripped as furniture and any page that didn't cleanly split, for the eyes-on gate.
10. **Full-recall scope-prefilter — the retrieval query must never silently under-return eligible chunks (catch 2).** A plain `WHERE …scope… ORDER BY embedding <=> q LIMIT k` against HNSW post-filters ANN candidates and can drop eligible chunks — unacceptable for a legal-weight answer. Make **full recall of the eligible set** the requirement: either use a **flat/exact scan** with the scope `WHERE` (correct by construction; almost certainly fast enough at this corpus size — prefer this and note whether HNSW is even warranted yet), or pgvector **iterative index scan**. Add a **harness assertion** that a known-eligible chunk is always returned for a query that should surface it (i.e. the ANN layer never drops an eligible chunk). The scope prefilter's correctness is never traded for ANN speed.
